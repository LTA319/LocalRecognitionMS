**版本**: 1.0

## 1. 引言与总体设计

### 1.1 系统概述

物料识别系统是一个基于 YOLO ONNX 模型的本地桌面应用，通过摄像头实时预览，支持手动或定时抓帧，对画面中的物料进行目标检测，并将识别结果映射到本地物料数据库，展示名称、规格、库存等详细信息。

核心流程：摄像头预览 → 抓帧 → 模型推理 → 分层匹配引擎查询 → 结果展示。

### 1.2 术语表

| 术语 | 说明 |
|------|------|
| YOLO | You Only Look Once，实时目标检测模型，本项目使用 YOLOv8n/YOLO11n nano 变体 |
| ONNX | Open Neural Network Exchange，跨平台模型交换格式，由 ONNX Runtime 执行推理 |
| FTS5 | SQLite 全文搜索引擎，支持前缀匹配和排序 |
| 别名映射 | 将 YOLO 输出的检测名称转换为数据库标准名称的对照表 |
| Embedding | 文本向量嵌入，用于语义相似度匹配 |

### 1.3 总体架构

系统采用分层架构，自下而上分为三层：

```text
ObjectRecognitionSystem/
├── Forms/                   # 表示层 — WinForms UI
│   └── MainForm.cs
├── Services/                # 服务层 — 业务逻辑
│   ├── CameraService.cs     #   摄像头捕获
│   ├── DetectionService.cs  #   YOLO 推理
│   ├── MatchingService.cs   #   分层匹配引擎
│   └── DatabaseService.cs   #   数据库操作
├── Models/                  # 数据模型
│   ├── DetectionResult.cs
│   ├── ItemInfo.cs
│   └── NameMapping.cs
├── Data/
│   └── AppDbContext.cs      # EF Core 上下文
├── Assets/
│   └── models/
│       └── yolo11n.onnx     # ONNX 模型文件
├── App.config               # 配置文件
└── Program.cs
```

### 1.4 技术选型

| 组件 | 选型 | 说明 |
|------|------|------|
| 运行时 | .NET 8 (WinForms) | 长期支持版本，性能优异 |
| 目标检测 | YoloDotNet | 纯 .NET 封装 YOLOv8+/ONNX，支持 CPU/CUDA/DirectML |
| 摄像头 | Emgu.CV | 功能全面，支持多种视频源 |
| 数据库 | SQLite + EF Core | 零配置本地文件数据库 |
| 字符串匹配 | FuzzySharp | FuzzyWuzzy 的 .NET 移植，支持 WeightedRatio 等多种算法 |
| 向量匹配 | Microsoft.ML / ONNX | 可选模块，语义相似度匹配 |
| 图像处理 | System.Drawing.Common | 图像缩放和格式转换 |

### 1.5 性能目标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 端到端响应 | ≤ 2 秒 | 从抓帧到结果展示 |
| 模型推理 | < 500ms | 单帧 640×640 推理 |
| 数据库查询 | < 100ms | 含匹配引擎流水线 |
| UI 帧率 | ≥ 25 FPS | 预览画面流畅 |

### 1.6 系统流程序列

```text
用户点击"识别"
    │
    ▼
CameraService.GrabFrame()          // 抓取当前帧 Bitmap
    │
    ▼
DetectionService.DetectAsync()     // YOLO ONNX 推理，返回 List<Detection>
    │
    ▼
MatchingService.ResolveAsync()     // 分层匹配引擎
    │
    ▼
UI 绑定结果显示                     // 物品名称、规格、价格、置信度
```

## 2. 模块分解与接口定义

### 2.1 表示层 — MainForm

**职责**: 提供摄像头预览、识别控制、结果展示与用户反馈入口。

**关键 UI 组件**:

| 组件 | 类型 | 用途 |
|------|------|------|
| pbPreview | PictureBox | 摄像头实时预览画面 |
| btnCapture | Button | 手动触发一次识别 |
| cbAutoDetect | CheckBox | 开启/关闭定时自动识别 |
| lblName / lblConfidence | Label | 显示识别结果和置信度 |
| dgvCandidates | DataGridView | 匹配候选列表，供用户选择纠正 |
| btnFeedback | Button | 用户手动纠正，写入别名映射表 |

**依赖**: CameraService、DetectionService、MatchingService（均由构造函数注入或 AppConfig 工厂创建）。

**自动识别速率**: 每 1.5 秒抓一帧，由 Timer 驱动，避免 CPU 过载。

### 2.2 服务层

#### 2.2.1 CameraService

**职责**: 封装摄像头生命周期管理与帧抓取。

```csharp
public class CameraService : IDisposable
{
    // 启动摄像头并绑定到预览控件
    public void StartCamera(PictureBox pbPreview);

    // 停止预览
    public void StopCamera();

    // 抓取当前帧（用于识别）
    public Bitmap? GrabFrame();

    // 释放摄像头资源
    public void Dispose();
}
```

**实现要点**:
- 使用 Emgu.CV `VideoCapture` 打开默认摄像头（index=0）
- 预览用 30ms Timer（~30FPS），每次释放旧 `Image` 防止内存泄漏
- `GrabFrame()` 从当前流抓一帧，调用方负责 Dispose 返回的 Bitmap
- `StartCamera` / `StopCamera` 内部管理 Timer 启停

**依赖**: Emgu.CV、System.Windows.Forms（PictureBox）

---

#### 2.2.2 DetectionService

**职责**: 加载 ONNX 模型，执行目标检测推理。

```csharp
public class DetectionService : IDisposable
{
    // 构造函数完成模型加载与预热
    public DetectionService(YoloOptions options);

    // 异步检测，返回所有检测结果
    public Task<List<DetectionResult>> DetectAsync(Bitmap image);

    // 程序启动时调用，确保模型已加载
    public void WarmUp();

    // 释放模型资源
    public void Dispose();
}
```

**DetectionResult 模型**:

```csharp
public class DetectionResult
{
    public string Name { get; set; }         // 检测标签（如 "bottle"）
    public double Confidence { get; set; }   // 置信度 [0, 1]
    public Rectangle BoundingBox { get; set; }
}
```

**实现要点**:
- 内部使用 `YoloDotNet.Yolo` 对象，`DetectAsync` 用 `Task.Run` 包装同步调用
- 预热时跑一次空图推理，触发 ONNX Runtime 首次编译
- 图像在推理前自动 resize 到模型输入尺寸（640×640）
- 支持 CUDA / DirectML 加速，通过 `YoloOptions.Cuda` 配置

**依赖**: YoloDotNet、System.Drawing

---

#### 2.2.3 MatchingService

**职责**: 分层匹配引擎，将检测结果映射到数据库物品。

```csharp
public class MatchingService
{
    public MatchingService(DatabaseService dbService);

    // 主入口：识别后调用，返回最佳匹配
    public Task<MatchResult?> ResolveAsync(string detectedName, double confidence);
}

public class MatchResult
{
    public ItemInfo Item { get; set; }
    public string MatchMethod { get; set; }  // "Mapping" | "Exact" | "FTS5" | "Fuzzy" | "Semantic"
    public double MatchScore { get; set; }
    public List<ItemInfo> Candidates { get; set; }  // 备选列表
}
```

**内部匹配流水线**（按优先级逐级降级）:

```text
Layer 1: 别名映射表（内存 Dictionary）
    ↓ 未命中
Layer 2: 精确匹配 + FTS5 全文搜索
    ↓ 未命中
Layer 3: FuzzySharp 字符串模糊匹配（WeightedRatio）
    ↓ 未命中
Layer 4: 向量语义匹配（可选，默认不启用）
    ↓
返回 Top-N 候选列表供用户选择
```

**各层实现细节**:

| 层级 | 方法 | 速度 | 说明 |
|------|------|------|------|
| L1 别名映射 | 内存 `ConcurrentDictionary<string, string>` 查找 | < 1ms | 启动时从 `NameMappings` 表加载到内存 |
| L2 精确+FTS5 | `EF.Functions.Like` + `ItemsFTS MATCH` | < 10ms | FTS5 支持前缀通配符 |
| L3 FuzzySharp | `Fuzz.WeightedRatio()` + `PartialRatio()` | < 100ms | 所有物品已在内存缓存，并行计算 |
| L4 语义向量 | ONNX Embedding 模型 + 余弦相似度 | ~200ms | 可选模块，需额外 ONNX 模型 |

**用户纠正流程**: 当用户在候选列表中选择了一个非默认匹配项，系统将 `(detectedName → selectedStandardName)` 写入 `NameMappings` 表并刷新内存缓存。

**依赖**: DatabaseService、FuzzySharp、（可选）Microsoft.ML

---

#### 2.2.4 DatabaseService

**职责**: 封装 SQLite 数据访问，为 MatchingService 和 UI 提供查询入口。

```csharp
public class DatabaseService
{
    // 精确名称查询
    public Task<ItemInfo?> GetByNameAsync(string name);

    // FTS5 全文搜索
    public Task<List<ItemInfo>> SearchFtsAsync(string keyword, int top = 5);

    // 模糊搜索（Like）
    public Task<List<ItemInfo>> FuzzySearchAsync(string keyword, int top = 10);

    // 获取所有物品（用于内存缓存预热）
    public Task<List<ItemInfo>> GetAllItemsAsync();

    // 别名映射管理
    public Task<string?> GetMappedNameAsync(string detectedName);
    public Task AddOrUpdateMappingAsync(string detectedName, string standardName);

    // 确保数据库已创建
    public void EnsureCreated();
}
```

**实现要点**:
- 内部持有 `AppDbContext` 实例，所有方法返回 `Task<>`（async/await）
- 别名映射在构造函数中全量加载到 `Dictionary<string, string>`，`GetMappedNameAsync` 返回 `Task.FromResult`
- FTS5 查询使用 `FromSqlRaw` 拼接 MATCH 语句

**依赖**: EF Core、SQLite

### 2.3 数据层

#### 2.3.1 实体模型

**ItemInfo**:

```csharp
public class ItemInfo
{
    public int Id { get; set; }
    public string Name { get; set; }          // 数据库标准名称（主匹配键）
    public string DisplayName { get; set; }   // 展示名称
    public string Code { get; set; }          // 货号 / 型号 / 条码
    public string Category { get; set; }       // 分类
    public string Specification { get; set; }  // 规格
    public decimal Price { get; set; }
    public int Stock { get; set; }
    public string Description { get; set; }
    public string ImagePath { get; set; }      // 物品图片路径（可选）
}
```

**NameMapping**:

```csharp
public class NameMapping
{
    public int Id { get; set; }
    public string DetectedName { get; set; }   // YOLO 输出的标签
    public string StandardName { get; set; }   // 映射到的标准名称
    public double Confidence { get; set; } = 1.0;  // 映射权重
    public string Category { get; set; }
}
```

#### 2.3.2 数据库上下文

```csharp
public class AppDbContext : DbContext
{
    public DbSet<ItemInfo> Items { get; set; }
    public DbSet<NameMapping> NameMappings { get; set; }

    protected override void OnConfiguring(DbContextOptionsBuilder options)
        => options.UseSqlite(AppConfig.ConnectionString);

    protected override void OnModelCreating(ModelBuilder model)
    {
        // 配置索引
        model.Entity<ItemInfo>().HasIndex(i => i.Name);
        model.Entity<ItemInfo>().HasIndex(i => i.Code);
        model.Entity<NameMapping>().HasIndex(m => m.DetectedName);
    }
}
```

#### 2.3.3 FTS5 虚拟表

```sql
CREATE VIRTUAL TABLE IF NOT EXISTS ItemsFTS USING fts5(
    Name, Code, Specification, DisplayName,
    content='Items', content_rowid='Id'
);

-- 触发器：Items 表变更时自动同步 FTS5 索引
CREATE TRIGGER IF NOT EXISTS items_ai AFTER INSERT ON Items BEGIN
    INSERT INTO ItemsFTS(rowid, Name, Code, Specification, DisplayName)
    VALUES (new.Id, new.Name, new.Code, new.Specification, new.DisplayName);
END;
-- (DELETE / UPDATE 触发器同理，略)
```

**约定**:

| ImagePath | TEXT | | 物品图片路径（本地文件） |

#### NameMappings 表

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| Id | INTEGER | PK, AUTOINCREMENT | 主键 |
| DetectedName | TEXT | NOT NULL, INDEX | YOLO 输出标签 |

```sql
CREATE VIRTUAL TABLE ItemsFTS USING fts5(
    Name, Code, Specification, DisplayName,
    content='Items',
    content_rowid='Id'
);
```

索引覆盖字段: `Name`, `Code`, `Specification`, `DisplayName`。

### 3.2 内存缓存

| 缓存项 | 数据结构 | 加载时机 | 容量 |
|--------|----------|----------|------|
| 别名映射 | `ConcurrentDictionary<string, string>` (检测名→标准名) | 启动时全量加载 | ~5K 条 |
| 物品缓存 | `ConcurrentDictionary<string, ItemInfo>` (标准名→物品) | 启动时全量加载 | ~50K 条 |
| 物品列表 | `List<ItemInfo>` | 启动时全量加载 | 供 FuzzySharp 遍历计算 |

**缓存刷新**: 用户纠正/新增映射后，调用 `DatabaseService.AddOrUpdateMappingAsync` 写入数据库，并同步更新内存 `ConcurrentDictionary`。

### 3.3 配置文件（App.config）

```xml
<?xml version="1.0" encoding="utf-8"?>
<configuration>
  <appSettings>
    <!-- SQLite 数据库文件路径（相对于程序目录或绝对路径） -->
    <add key="ConnectionString" value="Data Source=items.db" />

    <add key="OnnxModelPath" value="Assets/models/yolo11n.onnx" />

    <!-- 识别置信度阈值，低于此值的检测结果将被丢弃 -->

    <add key="PreviewFps" value="30" />

    <add key="SemanticMatchEnabled" value="false" />
```

## 4. 技术细节与约束

### 4.1 运行模式配置

| 模式 | 引擎 | CudaEnabled | 适用场景 |
|------|------|-------------|----------|
| CPU | ONNX Runtime CPU | false | 无 GPU 环境 |
| CUDA | ONNX Runtime GPU (NVIDIA) | true | 有 NVIDIA 显卡 |
| DirectML | ONNX Runtime DirectML | 需额外配置 | Windows 通用 GPU 加速 |

**默认**: CPU 模式。生产环境检测到 NVIDIA GPU 时可通过 App.config 切换 CUDA。

### 4.2 图像预处理流程

```text
摄像头原始帧 (BGR, 任意分辨率)
    → Emgu.CV 抓取 Mat
    → Mat.ToBitmap() 转换为 System.Drawing.Bitmap
    → YoloDotNet 内部 resize 到 640×640
    → 归一化（YoloDotNet 内部处理）
    → ONNX 推理
```

不额外做亮度/对比度增强，保留模型默认预处理管线。

### 4.3 帧率控制策略

| 场景 | 策略 |
|------|------|
| 预览 | 30FPS Timer 驱动，每次释放旧 Image 再赋值 |
| 手动识别 | 点击按钮瞬间抓一帧，不阻塞预览 |
| 自动识别 | 1.5 秒间隔 Timer 驱动，与预览 Timer 独立 |
| 连续识别保护 | `Interlocked` 标志位防重入：上一帧推理未完成时跳过本次触发 |

### 4.4 异步处理约定

- **所有 I/O 和推理操作使用 `async/await`**：数据库查询、模型推理均通过 `Task.Run` 或原生异步 API
- **UI 线程职责**: 只负责控件绑定和事件分发，不执行任何耗时操作
- **异常传播**: 服务层异常在 UI 层统一 catch，通过 `MessageBox` 提示用户，同时写入日志
- **取消支持**: 窗体关闭时通过 `CancellationTokenSource.Cancel()` 通知后台任务退出

### 4.5 错误处理与日志

| 场景 | 处理方式 |
|------|----------|
| 摄像头未找到 | 提示用户检查摄像头连接，禁止进入识别模式 |
| ONNX 模型加载失败 | 启动时检测文件是否存在，异常时弹窗并禁用识别按钮 |
| 推理返回空结果 | 显示"未检测到物品，请调整角度/光照" |
| 匹配引擎未命中 | 显示候选列表让用户选择；无候选时提示"未知物品，点此添加" |
| 数据库连接失败 | 启动时 fail-fast，弹窗后退出 |

**日志**: 使用 `Microsoft.Extensions.Logging` + 文件 provider，记录：识别时间、检测名、匹配结果、耗时分段。日志路径: `logs/app-{yyyyMMdd}.log`。

### 4.6 启动顺序

```text
1. 显示启动窗口 / 进度条
2. 加载 App.config
3. 初始化 DatabaseService（EnsureCreated + 预热缓存）
4. 初始化 DetectionService（加载 ONNX + WarmUp 空图推理）
5. 初始化 CameraService（不自动启动）
6. 启动 MainForm，用户手动打开摄像头
```

### 4.7 资源释放

- `MainForm.FormClosing` 中依次调用: `CameraService.StopCamera()` → `CameraService.Dispose()` → `DetectionService.Dispose()` → `DbContext.Dispose()`
- `DetectionService` 和 `CameraService` 均需实现 `IDisposable`

## 5. 风险与约束

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| COCO 预训练模型对特定物料准确率低 | 识别结果不可用 | 自定义数据集训练，单类 mAP > 0.85 |
| 5 万条数据 FuzzySharp 全量计算 | 匹配耗时突破 100ms | 先走 L1/L2 快速路径；L3 可并行化 |
| ONNX 模型加载导致首次启动慢 | 用户体验差 | 异步加载 + 进度条；WarmUp 预热 |
| 中英文混合标签 | 匹配歧义 | 别名映射表覆盖常见差异 |
| 摄像头权限被拒 | 功能不可用 | 启动时检测，友好提示用户授权 |
